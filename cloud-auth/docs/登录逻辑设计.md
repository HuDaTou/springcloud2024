# cloud-auth 后端登录逻辑设计

## 概述

cloud-auth 是整个系统的认证授权中心，提供用户注册、账号密码登录、注销等接口。前端（博客前台 / 后台管理）通过 HTTP 请求与 cloud-auth 交互完成身份认证。

---

## 一、涉及文件清单

| 文件 | 职责 |
|---|---|
| `config/SecurityConfig.java` | 安全过滤器链配置（登录/注销端点、过滤器顺序） |
| `filter/JsonUsernamePasswordAuthenticationFilter.java` | JSON 登录过滤器，解析 `application/json` 请求体 |
| `filter/LoginSuccessHandler.java` | 登录成功处理器，生成 JWT Token 并返回 |
| `filter/LoginFailureHandler.java` | 登录失败处理器，映射异常类型为业务错误信息 |
| `controller/AuthController.java` | 认证控制器（注册、权限码查询） |
| `service/AuthService.java` | 认证服务接口 |
| `service/impl/AuthServiceImpl.java` | 认证服务实现（注册业务逻辑） |
| `service/UserService.java` | 用户服务接口（含 `loadUserByUsername`） |
| `service/impl/UserServiceImpl.java` | 用户服务实现（登录时按用户名/邮箱查询用户） |
| `entity/PO/LoginUser.java` | Spring Security 认证用户对象（实现 `UserDetails`） |
| `entity/DTO/UserRegisterDTO.java` | 注册请求 DTO（含参数校验） |

## 二、登录流程

### 2.1 请求端点

```
POST /auth/login
Content-Type: application/json
```

### 2.2 请求体格式

```json
{
    "username": "admin",
    "password": "123456"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `username` | string | 是 | 用户名或邮箱（支持两者登录） |
| `password` | string | 是 | 密码明文 |

### 2.3 服务端处理流程

```
请求到达 POST /auth/login
  │
  ▼
JsonUsernamePasswordAuthenticationFilter.attemptAuthentication()
  │ 检测 Content-Type: application/json
  │ 解析 JSON 请求体，提取 username 和 password
  │ 构造 UsernamePasswordAuthenticationToken
  ▼
AuthenticationManager.authenticate()
  │ 委托给 UserServiceImpl.loadUserByUsername(username)
  ▼
UserServiceImpl.loadUserByUsername(usernameOrEmail)
  │ 按 username 或 email 查询数据库（OR 条件）
  │ 检查用户是否被禁用（isDisable）
  │ 检查用户是否被封禁（blacklist）
  │ 查询用户权限列表
  │ 返回 LoginUser(user, permissions)
  ▼
AuthenticationManager 验证密码
  │ PasswordEncoder.matches(rawPassword, encodedPassword)
  ▼
  成功 → LoginSuccessHandler.onAuthenticationSuccess()
  失败 → LoginFailureHandler.onAuthenticationFailure()
```

### 2.4 登录成功响应

```json
{
    "code": 200,
    "message": "登录成功",
    "data": {
        "accessToken": "eyJhbGciOiJ...",
        "expire": "2025-06-22T12:30:00Z"
    }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 200 表示成功 |
| `message` | string | "登录成功" |
| `data.accessToken` | string | JWT Token 字符串 |
| `data.expire` | string | Token 过期时间（ISO 8601 格式） |

**JWT Token 中包含的 Claims**：

| Claim | 说明 |
|---|---|
| `iss` | Token 签发者：`http://localhost:9123` |
| `sub` | 用户名 |
| `iat` | 签发时间 |
| `exp` | 过期时间（签发后 30 分钟） |
| `user_id` | 用户 ID |
| `username` | 用户名 |
| `authorities` | 用户权限码列表，如 `["auth:user:query", "auth:role:list"]` |

**Token 有效期**：30 分钟（`TOKEN_EXPIRE_MINUTES = 30`）

### 2.5 登录失败响应

```json
{
    "code": 500,
    "message": "用户名密码错误",
    "data": null
}
```

**异常映射表**：

| Spring Security 异常 | HTTP code | 错误信息 |
|---|---|---|
| `UsernameNotFoundException` | 500 | "用户不存在或已被禁用" |
| `BadCredentialsException` | 500 | "用户名密码错误" |
| `AccountExpiredException` | 500 | "账户已过期" |
| `CredentialsExpiredException` | 500 | "凭证已过期，请修改密码" |
| `DisabledException` | 403 | "账户已被禁用" |
| `LockedException` | 403 | "账户已被封禁" |

## 三、注销流程

### 3.1 请求端点

```
POST /auth/logout
```

无需请求体，只需携带有效的 Authorization Header。

### 3.2 服务端处理

```
请求到达 POST /auth/logout
  │ Spring Security 自动清除 SecurityContext
  │ 执行 logoutSuccessHandler
  ▼
  返回: {"code":200,"msg":"退出成功"}
```

### 3.3 注销响应

```json
{
    "code": 200,
    "msg": "退出成功"
}
```

**前端需要做的事**：
1. 调用 `POST /auth/logout`
2. 移除本地存储的 Token（localStorage / sessionStorage）
3. 跳转到首页或登录页

## 四、注册流程

### 4.1 请求端点

```
POST /auth/register
Content-Type: application/json
```

### 4.2 请求体格式

```json
{
    "username": "newuser",
    "password": "123456",
    "code": "a1B2c3",
    "email": "test@example.com"
}
```

| 字段 | 类型 | 必填 | 约束 | 说明 |
|------|------|------|------|------|
| `username` | string | 是 | 正则 `^[a-zA-Z0-9\u4e00-\u9fa5]+$`，长度 1-10 | 用户名（支持中英文数字） |
| `password` | string | 是 | 长度 6-20 | 明文密码，服务端 BCrypt 加密存储 |
| `code` | string | 是 | 长度固定 6 | 邮箱验证码 |
| `email` | string | 是 | 合法邮箱格式，长度 >= 4 | 电子邮箱 |

### 4.3 服务端处理流程

```
请求到达 POST /auth/register
  │ @Valid 参数校验
  ▼
AuthServiceImpl.register(UserRegisterDTO)
  │
  ├─ 1. 校验验证码:
  │     Redis Key: "register:code:" + email
  │     比对 Redis 中的验证码与请求中的 code
  │     → 为空/不存在/不匹配 → 返回失败
  │
  ├─ 2. 检查用户名和邮箱唯一性:
  │     按 username 或 email 查询数据库
  │     → 已存在 → 返回 "用户名或邮箱已存在"
  │
  ├─ 3. 创建用户:
  │     password = BCrypt.encode(rawPassword)
  │     registerType = 0 (邮箱注册)
  │     isDisable = false
  │     avatar = 默认头像 URL
  │
  ├─ 4. 分配默认角色:
  │     查询默认角色，插入 user_role 关联表
  │
  └─ 5. 返回成功（@Transactional 保证原子性）
```

### 4.4 注册响应

**成功**：
```json
{
    "code": 200,
    "message": "success",
    "data": null
}
```

**失败**：
```json
{
    "code": 500,
    "message": "用户名或邮箱已存在",
    "data": null
}
```

常见失败原因：
- "验证码不能为空" / "验证码不存在" / "验证码错误或已过期"
- "用户名或邮箱已存在"

## 五、前置依赖：发送验证码

### 5.1 请求端点

```
POST /auth/send-code
Content-Type: application/json
```

### 5.2 请求体

```json
{
    "email": "test@example.com",
    "type": "register"
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `email` | string | 接收验证码的邮箱 |
| `type` | string | `"register"` 注册 / `"reset"` 重置密码 |

### 5.3 响应

发送成功返回 `200`，验证码存入 Redis（带过期时间），前端无感知。

## 六、安全过滤器链

登录请求经过的安全过滤器链（按顺序）：

```
1. InternalServiceAuthenticationFilter    ← 内部 Feign 调用验证（X-Internal-Service 头）
2. JsonUsernamePasswordAuthenticationFilter ← JSON 登录处理（拦截 POST /auth/login）
3. BearerTokenAuthenticationFilter         ← JWT 资源服务器认证（保护其他接口）
```

**关键机制**：
- 登录接口 `/auth/login` 在公开白名单中，不要求 Bearer Token
- `JsonUsernamePasswordAuthenticationFilter` 支持 `application/json` 格式的登录请求
- 注册、发送验证码等接口也在公开白名单中

## 七、前端接入指引

### 7.1 登录接口调用

```
POST /auth/login
Body: {"username":"admin","password":"123456"}

成功 → 保存 data.accessToken 和 data.expire
     → 后续请求 Header: Authorization: Bearer <accessToken>
     → 调用 GET /user/auth/info 获取用户详情

失败 → 提示 data.message
```

### 7.2 注册接口调用

```
1. POST /auth/send-code  → 发送验证码到邮箱
   Body: {"email":"test@example.com","type":"register"}

2. POST /auth/register   → 提交注册
   Body: {"username":"newuser","password":"123456","code":"a1B2c3","email":"test@example.com"}

3. 成功后跳转到登录页
```

### 7.3 注销接口调用

```
POST /auth/logout
成功 → 清除本地 Token → 跳转首页
```

## 八、Token 在前端的处理要点

1. **Token 存储**：登录成功后前端应将 `accessToken` 和 `expire` 一起存储（localStorage 或 sessionStorage）
2. **Token 携带**：所有需要认证的请求在 `Authorization` 头中携带 `Bearer <token>`
3. **Token 过期**：前端存储时记录 `expire` 时间，读取时校验是否过期，过期则提示重新登录
4. **注销**：调用注销接口后必须清除本地 Token
