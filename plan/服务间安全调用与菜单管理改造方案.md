# 服务间安全调用与菜单管理改造方案

## 一、背景

### 1.1 改造前的问题

| 问题 | 描述 |
|---|---|
| 服务间调用无认证 | Feign 调用直接打到 Controller，无任何 Token 验证，依赖网关 IP 限制（不可靠） |
| InternalController 冗余（已解决） | `EmailInternalController`、`InternalBlackListController` 已删除，逻辑合并至各自的正常 Controller |
| 菜单接口缺失 | 前端需要 `GET /menu/all` 获取动态路由，后端无此接口 |
| 菜单缺少 CRUD | 仅有查询树形菜单，缺少新增/修改/删除 |
| cloud-auth 无 @EnableMethodSecurity | @PreAuthorize 注解不生效，权限形同虚设 |
| 注释/日志不规范 | 注释中英文混用，日志部分英文 |

## 三、核心设计

### 3.1 Token 体系

#### 用户 Token（authorization_code）

```json
{
  "sub": "1",
  "user_id": 1,
  "username": "admin",
  "authorities": ["auth:user:list", "auth:menu:list", "blog:black:add"]
}
```

#### 服务 Token（client_credentials）

```json
{
  "sub": "internal-service",
  "service": "internal-service",
  "authorities": ["ROLE_INTERNAL_SERVICE"]
}
```

**签发代码：** `CustomOAuth2TokenCustomizer.java`

```java
// 用户 Token
private void customizeUserToken(JwtEncodingContext context) {
    LoginUser loginUser = (LoginUser) context.getPrincipal().getPrincipal();
    context.getClaims().claim("user_id", loginUser.getUser().getId());
    context.getClaims().claim("username", loginUser.getUser().getUsername());
    context.getClaims().claim("authorities", loginUser.getAuthorities());
}

// 服务 Token
private void customizeClientCredentialsToken(JwtEncodingContext context) {
    String clientId = context.getRegisteredClient().getClientId();
    context.getClaims().claim("service", clientId);
    context.getClaims().claim("authorities", List.of("ROLE_INTERNAL_SERVICE"));
}
```

### 3.2 双层放行机制

```
请求进入
  │
  ├─ 第一层：URL 级（SecurityConfig）
  │   ├─ /** → internalServiceBypass()  ← 服务 Token → 直接放行，不再检查后续
  │   │    └─ 返回 null → 继续后续规则
  │   ├─ /oauth2/**, /auth/login... → permitAll()
  │   └─ anyRequest() → authenticated()
  │
  └─ 第二层：方法级（MethodSecurityConfig）
      └─ @PreAuthorize("hasAuthority('auth:user:list')")
          └─ InternalServiceSecurityExpressionRoot
              └─ 检测 ROLE_INTERNAL_SERVICE → return true
```

### 3.3 Feign 自动获取 Token 流程

```
微服务 A 启动
  → FeignClientCredentialsInterceptor 自动调用 POST /oauth2/token
    (grant_type=client_credentials, Basic Auth: internal-service:internal-secret)
  → cloud-auth 验证客户端 → 签发服务 Token (含 ROLE_INTERNAL_SERVICE)
  → Interceptor 缓存 Token（过期前 60s 自动刷新）
  → 每次 Feign 调用自动附加 Authorization: Bearer <token>
  → cloud-auth 双层放行 → 通过
```

## 五、安全模型

```
┌─────────────────────────────────────────────────┐
│                   请求到达                         │
└─────────────────────┬───────────────────────────┘
                      │
              ┌───────▼────────┐
              │  提取 JWT 并验签  │ ← oauth2ResourceServer
              │  解密 + 有效期    │
              └───────┬────────┘
                      │
              ┌───────▼────────┐
              │  提取 authorities│ ← JwtAuthenticationConverter
              │  注入 SecurityCtx│
              └───────┬────────┘
                      │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
  服务 Token     用户 Token     未认证
  ROLE_INTERNAL  auth:user:*    401
        │             │
        ▼             ▼
   全放行         按 @PreAuthorize
  任何接口       细粒度检查
```

## 六、配置说明

### 6.1 各微服务 application.yml

```yaml
cloud:
  internal:
    client-id: internal-service
    client-secret: internal-secret
    issuer-uri: http://localhost:9123   # cloud-auth 地址
```

### 6.2 cloud-auth SecurityConfig

```java
// 第一优先级：服务 Token 全放行
.requestMatchers("/**").access(internalServiceBypass())
// 其余规则...
.anyRequest().authenticated()
```
