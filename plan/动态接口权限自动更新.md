# 动态接口权限自动更新方案

## 一、方案概述

本方案实现了 Spring Cloud 微服务架构下的**接口权限自动扫描与注册**机制。当项目更新、新增接口后，无需手动在数据库中逐条添加权限记录，服务启动时会自动完成权限的收集、上报和持久化。

---

## 三、核心组件

| 组件 | 路径 | 职责 |
|------|------|------|
| **PermissionScanner** | `system-auth-starter/service/PermissionScanner.java` | 扫描 Controller，提取权限信息 |

## 四、使用方式

### 4.1 给接口添加权限注解

在 Controller 方法上使用 `@PreAuthorize` 注解：

```java
@RestController
@RequestMapping("/user")
public class UserController {

    /**
     * 用户列表
     * 权限标识: user:list
     */
    @GetMapping("/list")
    @PreAuthorize("hasAuthority('user:list')")
    @Operation(summary = "用户列表")
    public ResultData<List<UserVO>> listUsers() {
        return userService.listUsers();
    }

    /**
     * 新增用户
     * 权限标识: user:add
     */
    @PostMapping("/add")
    @PreAuthorize("hasAuthority('user:add')")
    @Operation(summary = "新增用户")
    public ResultData<Void> addUser(@RequestBody UserCreateDTO dto) {
        return userService.createUser(dto);
    }

    /**
     * 编辑用户
     * 权限标识: user:edit
     */
    @PutMapping("/edit")
    @PreAuthorize("hasAuthority('user:edit')")
    @Operation(summary = "编辑用户")
    public ResultData<Void> editUser(@RequestBody UserUpdateDTO dto) {
        return userService.updateUser(dto.getId(), dto);
    }

    /**
     * 删除用户
     * 权限标识: user:delete
     */
    @DeleteMapping("/delete/{id}")
    @PreAuthorize("hasAuthority('user:delete')")
    @Operation(summary = "删除用户")
    public ResultData<Void> deleteUser(@PathVariable Long id) {
        return userService.deleteUser(Collections.singletonList(id));
    }
}
```

### 4.2 权限标识命名规范

推荐格式：`模块:操作`

| 示例 | 说明 |
|------|------|
| `user:list` | 用户模块 - 查询列表 |
| `user:add` | 用户模块 - 新增 |
| `user:edit` | 用户模块 - 编辑 |
| `user:delete` | 用户模块 - 删除 |
| `order:export` | 订单模块 - 导出 |
| `system:config:edit` | 系统配置模块 - 编辑（三级） |

### 4.3 部署流程

```
1. 开发代码，给新接口添加 @PreAuthorize("hasAuthority('xxx')")
           ↓
2. 打包部署，服务重启
           ↓
3. PermissionSender 自动扫描并发送权限
           ↓
4. PermissionListener 接收并保存到 sys_permission 表
           ↓
5. 在权限管理后台给角色分配新权限
           ↓
6. 完成
```

## 五、技术实现细节

### 5.1 权限扫描器（PermissionScanner）

**核心逻辑**：

1. 获取所有 `@RestController` 标记的 Bean
2. 遍历每个 Controller 的所有方法
3. 检查方法是否有 HTTP 映射注解（`@GetMapping`、`@PostMapping` 等）
4. 解析 `@PreAuthorize("hasAuthority('xxx')")` 中的权限标识
5. 提取 OpenAPI 注解信息（`@Tag`、`@Operation`）
6. 组装完整的权限 DTO

**关键代码片段**：

```java
// 正则表达式，用于从 @PreAuthorize("hasAuthority('xxx')") 中提取 'xxx'
private static final Pattern PRE_AUTHORIZE_PATTERN = 
    Pattern.compile("hasAuthority\\('([^']+)'\\)");

private String getPermissionCode(Method method) {
    PreAuthorize preAuthorize = method.getAnnotation(PreAuthorize.class);
    if (preAuthorize != null) {
        String expression = preAuthorize.value();
        Matcher matcher = PRE_AUTHORIZE_PATTERN.matcher(expression);
        if (matcher.find()) {
            return matcher.group(1);  // 返回 xxx
        }
    }
    return null;  // 没有权限注解的接口不纳入管理
}
```

**扫描的数据结构**（PermissionDTO）：

| 字段 | 说明 | 来源 |
|------|------|------|
| serviceName | 服务名称 | `spring.application.name` |
| category | 接口分类 | `@Tag` 注解或类名 |
| name | 接口名称 | `@Operation(summary)` 或方法名 |
| code | 权限标识 | `@PreAuthorize` 中的 `hasAuthority('xxx')` |
| httpMethod | HTTP 方法 | `@GetMapping` 等 |
| path | 完整路径 | 服务前缀 + Controller 路径 + 方法路径 |

### 5.2 权限发送器（PermissionSender）

**实现接口**：`ApplicationRunner`（Spring Boot 启动后自动执行）

**核心逻辑**：

1. 服务启动完成后触发
2. 跳过 `test` 环境
3. 调用 `PermissionScanner.scanPermissions()` 获取权限列表
4. 序列化为 JSON
5. 通过 RabbitMQ 发送到 `auth.permission.register`

**关键代码片段**：

```java
@Override
public void run(ApplicationArguments args) throws Exception {
    // 跳过测试环境
    if (isTestProfile()) {
        return;
    }
    
    log.info("应用启动完成，开始扫描并注册权限信息...");
    List<PermissionDTO> permissions = permissionScanner.scanPermissions();

    if (!permissions.isEmpty()) {
        String permissionsJson = objectMapper.writeValueAsString(permissions);
        rabbitTemplate.convertAndSend(
            PERMISSION_EXCHANGE, 
            PERMISSION_REGISTER_ROUTING_KEY, 
            permissionsJson
        );
        log.info("已成功发送 {} 个权限点到 RabbitMQ", permissions.size());
    }
}
```

**RabbitMQ 配置**：

| 配置项 | 值 |
|--------|-----|
| Exchange | `auth.permission.exchange` |
| RoutingKey | `auth.permission.register` |
| 类型 | Topic Exchange |

### 5.3 权限监听器（PermissionListener）

**核心逻辑**：

1. 监听 `auth.permission.register.queue` 队列
2. 反序列化权限列表
3. 遍历每个权限，执行 `saveOrUpdate`
4. 根据 `permisson_code` 判断是新增还是更新

**关键代码片段**：

```java
private void saveOrUpdatePermission(PermissionDTO dto) {
    String code = dto.getCode();
    
    // 根据权限标识查询是否已存在
    SysPermission existing = permissionService.getOne(
        new LambdaQueryWrapper<SysPermission>()
            .eq(SysPermission::getPermissonCode, code)
    );

    if (existing != null) {
        // 已存在，检查字段是否需要更新
        boolean needUpdate = false;
        if (!dto.getName().equals(existing.getName())) {
            existing.setName(dto.getName());
            needUpdate = true;
        }
        if (!dto.getPath().equals(existing.getPath())) {
            existing.setPath(dto.getPath());
            needUpdate = true;
        }
        // ... 其他字段
        
        if (needUpdate) {
            existing.setUpdateTime(LocalDateTime.now());
            permissionService.updateById(existing);
            log.debug("更新权限: {} - {}", code, dto.getName());
        }
    } else {
        // 不存在，新增
        SysPermission newPermission = new SysPermission();
        newPermission.setPermissonCode(code);
        newPermission.setName(dto.getName());
        newPermission.setPath(dto.getPath());
        // ...
        permissionService.save(newPermission);
        log.debug("新增权限: {} - {}", code, dto.getName());
    }
}
```

### 5.4 自动配置（PermissionAutoConfiguration）

**条件化装配**：

```java
@Configuration
@ConditionalOnProperty(
    name = "permission.scanner.enabled", 
    havingValue = "true", 
    matchIfMissing = true  // 默认开启
)
public class PermissionAutoConfiguration {

    @Bean
    public PermissionScanner permissionScanner(ApplicationContext applicationContext) {
        return new PermissionScanner(applicationContext);
    }

    @Bean
    @ConditionalOnBean(RabbitTemplate.class)  // 仅当 RabbitMQ 可用时
    public PermissionSender permissionSender(...) {
        return new PermissionSender(...);
    }
}
```

**配置开关**：

```yaml
permission:
  scanner:
    enabled: true  # 开启权限扫描（默认true）
```

## 七、方案优势

### 7.1 自动化程度高

- ✅ 新增接口只需添加 `@PreAuthorize` 注解
- ✅ 部署重启自动扫描，无需手动操作
- ✅ 已存在的权限会自动更新字段

### 7.2 微服务友好

- ✅ 各微服务独立扫描，互不干扰
- ✅ 通过 RabbitMQ 异步解耦
- ✅ 支持服务名区分权限来源

### 7.3 可靠性强

- ✅ RabbitMQ 消息持久化，不丢失
- ✅ 幂等性设计（saveOrUpdate）
- ✅ 跳过测试环境，不污染数据

### 7.4 与 Spring Security 无缝集成

- ✅ 直接使用 Spring Security 的 `@PreAuthorize` 注解
- ✅ 权限校验由 Spring Security 原生实现
- ✅ 不需要自定义注解和拦截器

## 八、注意事项

### 8.1 必须添加 @PreAuthorize

```java
// ✅ 正确 - 会被扫描
@GetMapping("/list")
@PreAuthorize("hasAuthority('user:list')")
public ResultData listUsers() { ... }

// ❌ 错误 - 不会被扫描（权限不会入库）
@GetMapping("/list")
public ResultData listUsers() { ... }
```

### 8.2 权限标识唯一性

同一系统中，`permisson_code` 必须唯一。建议命名规范：

```
格式: {模块}:{操作}
示例: user:list, user:add, order:export, system:config:edit
```

### 8.3 权限与角色的关联

本方案只解决**权限自动入库**，**权限→角色**的分配仍需在后台管理界面手动配置。

如需进一步自动化，可以考虑：

- 新权限默认分配给超级管理员角色
- 自定义注解支持 `defaultRoles` 参数

## 十、总结

本方案通过以下三个环节实现了权限的自动管理：

1. **注解驱动**：使用 Spring Security 原生的 `@PreAuthorize` 注解
2. **启动扫描**：`ApplicationRunner` 在服务启动时自动扫描
3. **消息队列**：RabbitMQ 实现微服务间的异步权限上报

**适用场景**：Spring Cloud 微服务架构、周期性部署更新的项目。
