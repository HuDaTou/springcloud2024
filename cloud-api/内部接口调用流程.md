# 内部微服务接口调用流程

## 概述

本项目中的微服务之间通过 Feign 客户端进行内部调用。所有内部调用的鉴权基于 OAuth2 `client_credentials` 授权模式，持有 `ROLE_INTERNAL_SERVICE` 的服务 Token 拥有完全放行权限，不受 `@PreAuthorize` 细粒度权限控制限制。

---

## 整体架构

```
┌─────────────────────┐     Feign (带 client_credentials JWT)      ┌─────────────────────┐
│   调用方服务          │ ─────────────────────────────────────────→  │   被调用方服务        │
│  (cloud-web / any)  │                                              │  (cloud-auth / etc) │
└─────────────────────┘                                              └─────────────────────┘
         │                                                                      │
         │ 1. FeignInterceptor 请求 Token                                       │ 4. JwtAuthenticationConverter
         │    从 auth server 获取           ┌─────────────────────┐              │    提取 authorities
         └────────────────────────────────→ │   cloud-auth (9123)  │ ←─────────────┘
                                            │   授权服务器           │   5. SecurityConfig /
                                            │   OAuth2 /oauth2/token│      ResourceServerConfig
                                            └─────────────────────┘      URL 级 bypass
                                                    │                   6. InternalServiceExpressionRoot
                                                    │ 2. CustomOAuth2TokenCustomizer     方法级 bypass
                                                    │    注入 ROLE_INTERNAL_SERVICE
                                                    │    注入 service claim
                                                    ▼
                                              3. 返回 JWT
```

## 核心组件分工

### 一、cloud-api/config/ — Token 获取与注入

负责在 Feign 请求发起前自动获取服务 Token 并注入到请求头。

| 文件 | 角色 |
|------|------|
| `ClientCredentialsProperties.java` | 配置属性 `cloud.internal.*`，默认 client-id=`internal-service`，secret=`internal-secret`，issuer=`http://localhost:9123` |
| `FeignClientCredentialsConfig.java` | 创建 `FeignClientCredentialsInterceptor` Bean |
| `FeignClientCredentialsInterceptor.java` | Feign `RequestInterceptor`，在每次 Feign 调用前注入 `Authorization: Bearer <jwt>` |

**Token 获取逻辑**（`FeignClientCredentialsInterceptor`）：

```java
// 1. 检查本地缓存的 Token 是否有效（距离过期 > 60秒则有效）
// 2. 过期或不存在 → 向 {issuerUri}/oauth2/token 发送 HTTP POST
//    Authorization: Basic base64(client-id:client-secret)
//    body: grant_type=client_credentials
// 3. 解析响应中的 access_token，缓存起来
// 4. 注入到 Feign 请求 template 的 Authorization 头
```

存在该拦截器的 Feign 客户端（共 6 个）：

| Feign 客户端 | 目标服务 |
|-------------|---------|
| `AuthClient` | cloud-auth |
| `UserClient` | cloud-auth |
| `BlackListClient` | cloud-auth |
| `EmailClient` | cloud-auth |
| `MediaClient` | cloud-media-asset-processing |
| `FileUploadRulesClient` | cloud-media-asset-processing |

### 二、cloud-auth/config/ — 授权服务器端

负责签发服务 Token 并保护自身端点。

| 文件 | 角色 |
|------|------|
| `JweAuthorizationServerConfig.java` | OAuth2 授权服务器，注册 `internal-service` 客户端（`client_credentials` 模式） |
| `CustomOAuth2TokenCustomizer.java` | JWT 定制器，向 Token 注入 `ROLE_INTERNAL_SERVICE` |
| `SecurityConfig.java` | 安全过滤器链，包含 `internalServiceBypass()` |

**客户端注册**（`JweAuthorizationServerConfig`）：

```java
// 注册 client-id="internal-service"
// client-secret="{noop}internal-secret"
// grant_type = client_credentials
// scope = internal
```

**JWT 定制**（`CustomOAuth2TokenCustomizer`）：

```java
// client_credentials 模式下向 JWT 注入：
//   "service"      → 客户端 ID (如 "internal-service")
//   "authorities"  → ["ROLE_INTERNAL_SERVICE"]
```

### 三、cloud-system-starter/system-auth-starter/config/ — 被调用方服务端

以 Starter 方式通过 `spring.factories` 自动配置到所有下游微服务。

| 文件 | 角色 |
|------|------|
| `ResourceServerConfig.java` | 资源服务器安全配置，URL 级放行 `ROLE_INTERNAL_SERVICE` |
| `InternalServiceMethodSecurityExpressionHandler.java` | 自定义方法安全表达式处理器 |
| `InternalServiceSecurityExpressionRoot.java` | 代理模式拦截 `hasAuthority`/`hasRole` 等 SpEL 函数 |
| `JwtAuthenticationConverterConfig.java` | JWT 到 `Authentication` 的转换器，提取 `authorities` claim |
| `SecurityConst.java` | 定义 `ROLE_INTERNAL_SERVICE = "ROLE_INTERNAL_SERVICE"` |

## 完整调用流程

### 步骤 1：Feign 请求发起

```
cloud-web 服务通过 @FeignClient 调用 cloud-auth 的接口
→ FeignClientCredentialsInterceptor.apply()
→ 检查缓存的 Token 是否有效
```

### 步骤 2：获取 client_credentials Token

```
FeignClientCredentialsInterceptor 向 http://localhost:9123/oauth2/token 发送请求
→ Header: Authorization: Basic aW50ZXJuYWwtc2VydmljZTppbnRlcm5hbC1zZWNyZXQ=
→ Body:   grant_type=client_credentials
```

### 步骤 3：授权服务器处理

```
cloud-auth 的 JweAuthorizationServerConfig 验证客户端凭证
→ 认证通过，创建 JWT
→ CustomOAuth2TokenCustomizer.customizeClientCredentialsToken():
    claims.put("service", "internal-service")
    claims.put("authorities", List.of("ROLE_INTERNAL_SERVICE"))
→ 返回 JSON: { access_token: "eyJ...", expires_in: 3600 }
```

### 步骤 4：Token 注入 Feign 请求

```
FeignClientCredentialsInterceptor 缓存 Token（expires_at - 60s）
→ 设置 Feign requestTemplate header:
    Authorization: Bearer eyJhbG...(含 ROLE_INTERNAL_SERVICE 的 JWT)
```

### 步骤 5：被调用方接收请求（URL 级鉴权）

```
请求到达 cloud-auth 的 SecurityConfig.defaultSecurityFilterChain
→ authorizeHttpRequests 的第一条规则: internalServiceBypass()
→ 从 Authentication 中查找 ROLE_INTERNAL_SERVICE
→ 找到 → 返回 AuthorizationDecision(true) → 直接放行
→ 未找到 → 返回 null → 继续后续规则匹配
```

### 步骤 6：被调用方方法级鉴权

```
请求到达 Controller 方法，如 @PreAuthorize("hasAuthority('auth:role:list')")
→ InternalServiceMethodSecurityExpressionHandler 创建 InternalServiceSecurityExpressionRoot
→ hasAuthority("auth:role:list") 被拦截:
    1. 先检查 Authentication 是否有 ROLE_INTERNAL_SERVICE
    2. 有 → 直接 return true（绕过细粒度权限检查）
    3. 没有 → 委托给真实的 SecurityExpressionRoot 进行正常权限检查
```

---

## 两层安全防护

| 层级 | 机制 | 位置 |
|------|------|------|
| **URL 层** | `internalServiceBypass()` 检查 `ROLE_INTERNAL_SERVICE`，有则直接授权 | `SecurityConfig` / `ResourceServerConfig` |
| **方法层** | `InternalServiceSecurityExpressionRoot` 代理 Spring Security 的 `hasAuthority`/`hasRole` 等 SpEL 函数，持有 `ROLE_INTERNAL_SERVICE` 时全部返回 `true` | `InternalServiceMethodSecurityExpressionHandler` |

> `@ConditionalOnMissingBean(name = "defaultSecurityFilterChain")` 确保 `ResourceServerConfig`（Starter）不会覆盖 cloud-auth 自身的 `SecurityConfig`，cloud-auth 使用自己的安全过滤器链。

## 各服务配置

每个需要调用其他服务的微服务都需在 `application.yml` 中配置：

```yaml
cloud:
  internal:
    client-id: internal-service
    client-secret: internal-secret
    issuer-uri: http://localhost:9123   # cloud-auth 地址
```

## 关键设计要点

1. **client_credentials 模式**：服务之间的调用不涉及用户，使用 OAuth2 的机器对机器授权模式
2. **Token 缓存**：拦截器中缓存 Token，距离过期 60 秒前复用，避免每次请求都去授权服务器换 Token
3. **双重旁路**：URL 级 + 方法级双重放行 `ROLE_INTERNAL_SERVICE`，确保内部调用不受任何权限限制
4. **代理模式**：`InternalServiceSecurityExpressionRoot` 使用代理而非继承（因 Spring Security 6 中 `hasAuthority` 等方法为 `final`），在自身处理逻辑后委托给真实 root
5. **自动化配置**：`ResourceServerConfig` 通过 `spring.factories` 自动注册，所有引入
