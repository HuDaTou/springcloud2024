# 内部微服务接口调用流程

## 概述

本项目中的微服务之间通过 Feign 客户端进行内部调用。内部调用的鉴权基于简单的**请求头 + 共享密钥**机制：调用方在 Feign 请求头中注入 `X-Internal-Service` 和 `X-Service-Key`，被调用方验证后创建 `ROLE_INTERNAL_SERVICE` 认证对象，绕过所有权限检查。

---

## 整体架构

```
┌──────────────────────────────────────────────────────────────────────────────┐
│                           cloud-gateway (9527)                                │
│  RemoveInternalHeadersFilter: 剥离 X-Internal-Service / X-Service-Key 请求头  │
│  防止外部客户端伪造内部服务头绕过安全校验                                       │
└────────────────────┬─────────────────────────────────────────────────────────┘
                     │ 外部请求（请求头已被剥离）
                     ▼
┌──────────────────────────────────────────────────────────────────────────────┐
│                         被调用方微服务 (cloud-auth / cloud-web / etc)           │
│                                                                               │
│  InternalServiceAuthenticationFilter (Before JWT Filter)                      │
│    → 检测 X-Internal-Service + X-Service-Key                                  │
│    → 验证通过 → 创建 InternalServiceAuthentication(ROLE_INTERNAL_SERVICE)      │
│                                                                               │
│  internalServiceBypassAuthorizationManager → URL 级别放行                      │
│  InternalServiceMethodSecurityExpressionHandler → 方法级别 @PreAuthorize 放行  │
└──────────────────────────────────────────────────────────────────────────────┘
                     ▲
                     │ Feign 直连（不经网关，保留请求头）
┌────────────────────┴─────────────────────────────────────────────────────────┐
│                           调用方微服务 (cloud-web / etc)                        │
│                                                                               │
│  FeignInternalServiceInterceptor (RequestInterceptor)                         │
│    → 每次 Feign 调用自动注入:                                                  │
│        X-Internal-Service: true                                               │
│        X-Service-Key: <共享密钥>                                               │
└──────────────────────────────────────────────────────────────────────────────┘
```

## 核心组件分工

### 一、cloud-api/config/ — 请求头注入（调用方）

负责在 Feign 请求发起前自动注入内部服务标识头和共享密钥。

| 文件 | 角色 |
|------|------|
| `InternalServiceProperties.java` | 配置属性 `cloud.internal.service-key`，默认值 `internal-service-key-2024` |
| `FeignInternalServiceConfig.java` | 自动配置类，创建 `FeignInternalServiceInterceptor` Bean |
| `FeignInternalServiceInterceptor.java` | Feign `RequestInterceptor`，在每次 Feign 调用前注入两个请求头 |

**请求头注入逻辑**（`FeignInternalServiceInterceptor`）：

```java
// 每次 Feign 调用前执行
template.header("X-Internal-Service", "true");
template.header("X-Service-Key", serviceKey);  // 从配置读取的共享密钥
```

各微服务在 `application.yml` 中配置：

```yaml
cloud:
  internal:
    service-key: your-shared-key   # 所有微服务必须使用相同的密钥
```

### 二、cloud-system-starter/system-auth-starter/ — 请求验证与服务端放行（被调用方）

以 Starter 方式通过 `AutoConfiguration.imports` 自动配置到所有下游微服务。

#### 2.1 通用自动配置（对所有服务生效）

| 文件 | 角色 |
|------|------|
| `InternalServiceAuthAutoConfiguration.java` | 通用自动配置，提供 Filter 和 URL Bypass 两个共享 Bean |
| `InternalServiceAuthentication.java` | 内部服务认证对象，携带 `ROLE_INTERNAL_SERVICE` 权限 |
| `InternalServiceAuthenticationFilter.java` | 请求头验证过滤器，验证密钥并创建认证对象 |
| `InternalServiceMethodSecurityExpressionHandler.java` | 自定义方法安全表达式处理器 |
| `InternalServiceSecurityExpressionRoot.java` | 代理模式拦截 `hasAuthority`/`hasRole` 等 SpEL 函数 |

**`InternalServiceAuthAutoConfiguration` 提供的 Bean**：

```java
// 1. InternalServiceAuthenticationFilter — 验证 X-Internal-Service + X-Service-Key
// 2. internalServiceBypassAuthorizationManager — URL 级别绕过策略
```

#### 2.2 资源服务器安全配置（非 auth 服务自动生效）

| 文件 | 角色 |
|------|------|
| `ResourceServerConfig.java` | OAuth2 资源服务器配置，注入共享的 Filter 和 Bypass |

`@ConditionalOnMissingBean(name = "defaultSecurityFilterChain")` 确保 `ResourceServerConfig` 不会覆盖 cloud-auth 自身的 `SecurityConfig`。

**过滤器链顺序**：

```
InternalServiceAuthenticationFilter → BearerTokenAuthenticationFilter
```

内部服务验证在 JWT 验证之前执行，内部调用直接从请求头认证，不经过 OAuth2。

### 三、cloud-auth/config/ — 授权服务器安全

cloud-auth 有自己的安全配置，不使用 `ResourceServerConfig`，但注入共享组件。

| 文件 | 角色 |
|------|------|
| `SecurityConfig.java` | 安全过滤器链，注入共享的 `InternalServiceAuthenticationFilter` 和 `internalServiceBypassAuthorizationManager` |

### 四、cloud-gateway — 安全边界

| 文件 | 角色 |
|------|------|
| `RemoveInternalHeadersFilter.java` | 全局过滤器（`HIGHEST_PRECEDENCE`），对所有外部请求剥离 `X-Internal-Service` 和 `X-Service-Key` |

## 完整调用流程

### 步骤 1：Feign 请求发起

```
cloud-web 服务通过 @FeignClient 调用 cloud-auth 的接口
→ FeignInternalServiceInterceptor.apply()
→ 注入请求头:
    X-Internal-Service: true
    X-Service-Key: internal-service-key-2024
```

### 步骤 2：请求到达被调用方

```
请求到达 cloud-auth（Feign 直连，绕过网关，请求头完整保留）
```

### 步骤 3：InternalServiceAuthenticationFilter 验证

```
InternalServiceAuthenticationFilter.doFilterInternal()
→ 检测 X-Internal-Service == "true"
→ 验证 X-Service-Key == 配置的密钥
→ 匹配成功 → 创建 InternalServiceAuthentication(ROLE_INTERNAL_SERVICE)
→ 放入 SecurityContextHolder
```

### 步骤 4：URL 级别放行

```
SecurityConfig / ResourceServerConfig 的 authorizeHttpRequests:
→ 第一条规则: .requestMatchers("/**").access(internalServiceBypassAuthorizationManager)
→ 从 Authentication 中查找 ROLE_INTERNAL_SERVICE
→ 找到 → 返回 AuthorizationDecision(true) → 直接放行
→ 未找到 → 返回 null → 继续后续规则匹配（正常 JWT 认证流程）
```

### 步骤 5：方法级别放行

```
请求到达 Controller 方法，如 @PreAuthorize("hasAuthority('xxx')")
→ InternalServiceMethodSecurityExpressionHandler 创建 InternalServiceSecurityExpressionRoot
→ hasAuthority("xxx") 被拦截:
    1. 先检查 Authentication 是否有 ROLE_INTERNAL_SERVICE
    2. 有 → 直接 return true（绕过细粒度权限检查）
    3. 没有 → 委托给标准的 SecurityExpressionRoot 进行正常权限检查
```

### 步骤 6：网关安全边界

```
外部客户端请求 → cloud-gateway
→ RemoveInternalHeadersFilter (HIGHEST_PRECEDENCE):
    移除 X-Internal-Service 头
    移除 X-Service-Key 头
→ 下游微服务不会接收到这两个头（外部请求无法伪造内部调用）
```

---

## 两层安全防护

| 层级 | 机制 | 位置 |
|------|------|------|
| **网关层** | `RemoveInternalHeadersFilter` 剥离外部请求的内部服务头 | cloud-gateway |
| **URL 层** | `internalServiceBypassAuthorizationManager` 检查 `ROLE_INTERNAL_SERVICE` | `SecurityConfig` / `ResourceServerConfig` |
| **方法层** | `InternalServiceSecurityExpressionRoot` 代理 `hasAuthority`/`hasRole` 等 SpEL 函数 | `InternalServiceMethodSecurityExpressionHandler` |

## 各服务配置

每个微服务在 `application.yml` 中配置共享密钥：

```yaml
cloud:
  internal:
    service-key: internal-service-key-2024   # 所有微服务使用相同密钥
```

## 关键设计要点

1. **请求头 + 共享密钥**：不再依赖 OAuth2 client_credentials 获取 JWT，改为简单的请求头标记，减少对 auth 服务的依赖和网络开销
2. **Feign 直连绕过网关**：内部 Feign 调用直接连接目标服务，不经过 gateway，因此内部请求头不会被 `RemoveInternalHeadersFilter` 剥离
3. **网关安全边界**：`RemoveInternalHeadersFilter` 最高优先级剥离外部请求的内部服务头，外部客户端无法伪造
4. **双重旁路**：URL 级 + 方法级双重放行 `ROLE_INTERNAL_SERVICE`，确保内部调用不受任何权限限制
5. **通用组件共享**：`InternalServiceAuthAutoConfiguration` 提供 Filter 和 Bypass 策略的共享 Bean，cloud-auth 和所有下游服务复用同一套逻辑
6. **代理模式**：`InternalServiceSecurityExpressionRoot` 使用代理而非继承（因 Spring Security 6 中 `hasAuthority` 等方法为 `final`）
7. **过滤器顺序**：`InternalServiceAuthenticationFilter` 在 `BearerTokenAuthenticationFilter` 之前执行，内部调用直接从请求头认证，不依赖 JWT
